# Руководство администратора

### Настройка агента OpenHands и обработка результатов мониторинга LS Fusion

---

## 1. Назначение системы

Система состоит из двух ключевых компонентов:

- **Приложение в LS Fusion** — фиксирует критические события и отправляет HTTP-уведомления.
- **Агент на базе OpenHands** — принимает сигнал, запрашивает у большой языковой модели инструкцию и пересылает её администратору в мессенджер.

>  **Ключевой принцип:** агент не выполняет автоматических действий в LS Fusion. Он только доставляет рекомендацию. Все решения остаются за человеком.

---

##  2. Предварительные требования

Перед началом настройки подготовьте следующие компоненты:

| Компонент | Описание |
|-----------|----------|
|**Сервер** | Linux-машина с Python версии **3.10 и выше** |
| **API-ключ LLM** | Действующий ключ к OpenAI, YandexGPT, GigaChat или другой совместимой модели |
| **Telegram-бот** | Создаётся через [@BotFather](https://t.me/BotFather). Потребуются токен бота и `chat_id` получателя |
| **Matrix (опционально)** | Учётная запись бота, пароль и идентификатор комнаты |
| **GitLab-репозиторий** | Доступ к исходному коду, скриптам и конфигурациям проекта |
| **Сетевой доступ** | Открытый порт между LS Fusion и сервером, на котором работает агент |

---

##  3. Установка и начальная конфигурация

### 3.1. Клонирование репозитория

Склонируйте репозиторий из GitLab на целевой сервер.

### 3.2. Подготовка окружения

Запустите скрипт `setup_env.sh`. Он автоматически создаст виртуальное окружение Python и установит все необходимые зависимости, перечисленные в файле `requirements.txt`.

### 3.3. Заполнение конфигурации

Основные настройки агента хранятся в файле `config.yaml`. Администратору необходимо заполнить три секции:

**Секция `webhook`:**
- Порт, на котором агент будет принимать запросы от LS Fusion. По умолчанию используется порт `5000`.

**Секция `llm`:**
- Название провайдера (openai, yandex, gigachat).
- API-ключ для доступа к модели.
- Идентификатор используемой модели.

**Секция `messenger`:**
- Тип мессенджера: `telegram` или `matrix`.
- Для Telegram — токен бота и идентификатор чата.
- Для Matrix — адрес сервера, логин, пароль и ID комнаты.

---

##  4. Настройка LS Fusion

Приложение в LS Fusion необходимо доработать: при возникновении контрольного события оно должно отправлять HTTP-запрос на агента.

Параметры запроса:

| Параметр | Значение |
|----------|----------|
| **Метод** | `POST` |
| **URL** | `http://<IP-адрес-агента>:5000/event` |
| **Заголовок** | `Content-Type: application/json` |
| **Тело запроса** | JSON-объект с полем `description` |

Поле `description` должно содержать текстовое описание произошедшего события. Порт в URL должен совпадать с указанным в `config.yaml`.

>  **Важно:** убедитесь, что сервер агента доступен по сети с хоста LS Fusion. При необходимости настройте правила брандмауэра.

---

##  5. Запуск и проверка работоспособности

### 5.1. Тестовый запуск

Для первичной отладки запустите главный скрипт агента вручную из каталога `agent`. Все логи будут выводиться в консоль, что удобно для отслеживания ошибок.

### 5.2. Постоянный запуск

Для production-окружения настройте агента как systemd-сервис. Для этого:

- Создайте unit-файл с описанием сервиса.
- Укажите в нём путь к виртуальному окружению и главному исполняемому файлу.
- Настройте автоматический перезапуск при падении процесса.
- Зарегистрируйте и активируйте сервис.

### 5.3. Проверка сквозного прохождения

После запуска выполните полную проверку:

1. Инициируйте тестовое событие в LS Fusion.
2. Убедитесь, что агент принял запрос.
3. Проверьте, что агент отправил промпт в LLM и получил ответ.
4. Проконтролируйте доставку сообщения в мессенджер.

При возникновении ошибок анализируйте логи агента — в них фиксируются все этапы обработки.

---

##  6. Действия при получении сообщения

Когда администратор получает сообщение от агента, оно содержит текстовую инструкцию, сгенерированную LLM. Порядок действий:

| Шаг | Действие |
|:---:|----------|
| **1** | Оцените серьёзность ситуации на основе описания события и предложенной рекомендации |
| **2** | Примите решение о целесообразности выполнения предложенных действий |
| **3** | Выполните необходимые операции (проверки, корректировки, запуск процессов в LS Fusion, оповещение сотрудников) |
| **4** | Проконтролируйте результат. При необходимости повторите цикл |

>  **Помните:** агент — только канал доставки рекомендаций. Ответственность за принятие и исполнение решений полностью лежит на администраторе.

---

##  7. Мониторинг состояния агента

Для стабильной работы системы рекомендуется настроить регулярный контроль:

- **Доступность порта вебхука** — мониторинг с помощью любой системы наблюдения (Zabbix, Prometheus, простая проверка HTTP).
- **Логи агента** — периодически проверяйте их на наличие ошибок связи с LLM или мессенджером.
- **Срок действия токенов** — своевременно обновляйте API-ключи и токены ботов в `config.yaml` с последующей перезагрузкой сервиса.
- **Баланс платных API** — контролируйте лимиты и остаток средств в личном кабинете провайдера LLM.
