# LsFusion
Руководство администрирования

1. Назначение системы
Система состоит из приложения в LS Fusion и агента, созданного на базе платформы OpenHands. Агент отслеживает критические события, поступающие из LS Fusion, и при их возникновении запрашивает у большой языковой модели инструкцию о дальнейших действиях. Полученный ответ автоматически отправляется администратору в выбранный мессенджер — Telegram или Matrix. Таким образом, администратор оперативно получает рекомендации по реагированию на инциденты.

2. Предварительные требования
До начала настройки необходимо подготовить:

Сервер или виртуальную машину с операционной системой Linux и установленным Python версии не ниже 3.10.

Доступ к репозиторию GitLab, где хранятся все скрипты и конфигурационные файлы проекта.

Действующий API-ключ к выбранной LLM — OpenAI, YandexGPT, GigaChat или иной совместимой модели.

Учётную запись бота в мессенджере Telegram (создаётся через BotFather) и идентификатор чата, куда будут приходить уведомления. При использовании Matrix потребуется учётная запись пользователя-бота, пароль и идентификатор комнаты.

В LS Fusion должно быть настроено событие, по которому приложение отправляет HTTP-запрос на адрес агента.

3. Установка и начальная конфигурация
Первым шагом клонируйте репозиторий из GitLab на целевой сервер. После этого запустите скрипт подготовки окружения — он создаст виртуальное окружение Python и установит все необходимые зависимости согласно файлу requirements.txt.

Основные настройки системы хранятся в файле config.yaml. Администратор должен заполнить в нём следующие параметры:

В секции webhook при необходимости изменить порт, на котором агент будет принимать запросы от LS Fusion. По умолчанию используется порт 5000.

В секции llm указать провайдера, API-ключ и название используемой модели.

В секции messenger выбрать тип мессенджера и заполнить соответствующие поля: для Telegram — токен бота и идентификатор чата, для Matrix — адрес сервера, логин, пароль и идентификатор комнаты.

4. Настройка LS Fusion
Приложение в LS Fusion должно быть доработано таким образом, чтобы при возникновении контролируемого события выполнялся вызов внешнего HTTP-сервиса. Настройте действие со следующими параметрами:

Метод запроса — POST.

URL — http://<IP-адрес сервера агента>:5000/event. Порт должен совпадать с указанным в конфигурационном файле.

Заголовок Content-Type со значением application/json.

Тело запроса должно содержать JSON-объект с полем description, в которое помещается текстовое описание произошедшего события.

Важно обеспечить сетевую доступность сервера агента со стороны LS Fusion. При необходимости настройте правила брандмауэра и маршрутизацию.

5. Запуск и проверка работоспособности
Для тестового запуска агента выполните вручную главный скрипт агента из каталога agent. При этом в консоль будут выводиться все логи, что удобно для первичной отладки.

Для постоянной работы рекомендуется настроить агента как системный сервис операционной системы. Для этого необходимо создать и зарегистрировать systemd-юнит, указав в нём путь к виртуальному окружению и главному исполняемому файлу агента. Параметры юнита должны предусматривать автоматический перезапуск при падении процесса.

После запуска агента необходимо убедиться в корректности его работы. Инициируйте тестовое событие в LS Fusion и проверьте, что агент принял запрос, отправил промпт в LLM, получил ответ и переслал его в настроенный мессенджер. При возникновении ошибок следует анализировать логи агента — в них фиксируются все этапы обработки.

6. Порядок действий при получении сообщения
Когда администратор получает сообщение в мессенджере, оно содержит текстовую инструкцию от LLM. Сообщение следует воспринимать как рекомендацию по реагированию на зафиксированное событие. Администратор должен:

Оценить серьёзность ситуации на основе описания события и предложенной инструкции.

Выполнить рекомендованные действия, если они признаны целесообразными. Это может включать запуск дополнительных проверок в LS Fusion, ручную корректировку данных, уведомление других сотрудников, инициацию аварийных процедур и так далее.

Проконтролировать результат выполненных действий. При необходимости повторить цикл, вручную инициировав повторную обработку события.

Агент выполняет исключительно функцию доставки рекомендации и не предпринимает никаких автоматических действий в LS Fusion. Ответственность за принятие решений и их исполнение полностью лежит на администраторе.

7. Мониторинг состояния агента
Для контроля за работой агента рекомендуется:

Настроить мониторинг доступности порта вебхука с помощью любой системы мониторинга.

Периодически просматривать логи агента на предмет ошибок связи с LLM или мессенджером.

Проверять актуальность и срок действия API-ключей и токенов ботов, своевременно обновлять их в конфигурационном файле с последующей перезагрузкой сервиса.

При использовании платных API LLM следует также контролировать баланс и лимиты учётной записи, чтобы избежать сбоев из-за исчерпания квот.

8. Обновление и сопровождение
Исходные скрипты и документация хранятся в GitLab. Для обновления агента на продуктивном сервере необходимо получить актуальную версию из репозитория, при необходимости обновить зависимости и перезапустить сервис. Все изменения рекомендуется предварительно тестировать на стендовом окружении.
